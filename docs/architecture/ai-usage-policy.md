# AI Usage Policy

This document describes the AI usage policy layer that protects the LLM execution boundary in **Intelligent Automation Platform**.

## Purpose

The AI usage policy exists to prevent avoidable abuse and cost amplification before the platform reaches the agent executor.

It protects the system from:

- oversized inbound messages
- prompt envelopes that exceed token budgets
- excessive request rate per user and channel
- unnecessary LLM cost caused by runaway prompts

## Placement in the Architecture

The policy is evaluated inside the omnichannel orchestration flow immediately before agent execution.

Current runtime sequence:

```text
Orchestrator
  -> optional knowledge retrieval
  -> AI usage policy check
  -> Agent Executor
```

This position is intentional.

The policy evaluates the effective prompt envelope after any retrieved context has been assembled, which makes token estimation more realistic than validating only the raw inbound message.

## Current Rules

The policy currently validates:

- maximum inbound message size in characters
- maximum prompt token budget
- maximum completion token budget
- maximum AI requests per minute per user/channel combination

Current environment variables:

- `AI_MAX_PROMPT_TOKENS`
- `AI_MAX_COMPLETION_TOKENS`
- `MAX_MESSAGE_CHARACTERS`
- `MAX_AI_REQUESTS_PER_MINUTE`

## Decision Contract

`AiUsagePolicyService` returns a structured decision instead of pushing policy exceptions directly into the orchestrator flow.

Current shape:

```json
{
  "allow": true,
  "reason": null,
  "estimatedPromptTokens": 320,
  "maxPromptTokens": 4000,
  "maxCompletionTokens": 1000,
  "rateLimitKey": "ai:rate-limit:EMAIL:user-123"
}
```

When blocked, the service returns `allow: false` with one of the current reason codes:

- `MESSAGE_TOO_LONG`
- `TOKEN_LIMIT_EXCEEDED`
- `RATE_LIMIT_EXCEEDED`

## Orchestrator Behavior

When the policy allows the request:

1. the orchestrator records `ai_policy_checked`
2. the orchestrator records `ai_policy_passed`
3. agent execution proceeds normally

When the policy rejects the request:

1. the orchestrator records `ai_policy_checked`
2. the orchestrator records `ai_policy_rejected`
3. the agent executor is skipped
4. a safe response is generated by the platform
5. the response can still be dispatched through the normal outbound channel path

This keeps the user-facing behavior controlled while protecting the LLM boundary.

## Safe Rejection Behavior

Blocked requests do not attempt LLM execution.

Instead, the orchestrator returns a safe system-generated response explaining that the request exceeded platform limits, for example:

- message too large
- prompt too large
- rate limit reached

This behavior prevents accidental retries against the model while still giving clear feedback to the user.

## Rate Limiting

The current rate-control strategy uses the existing cache layer and builds a key from:

- channel
- sender ID, sender address, conversation ID, or fallback identifier

The effective key pattern is similar to:

```text
ai:rate-limit:{channel}:{identifier}
```

This keeps the implementation lightweight and compatible with the existing Redis-backed cache strategy.

## Observability

The policy extends the existing Prometheus instrumentation with:

- `ai_requests_total`
- `ai_requests_blocked_total`
- `ai_tokens_used_total`
- `ai_policy_rejections_total`

Rejection reasons are emitted as labels so blocked traffic can be analyzed separately from successful AI executions.

## Execution Timeline Integration

The policy is integrated with execution tracking and live activity.

Timeline events now include:

- `ai_policy_checked`
- `ai_policy_passed`
- `ai_policy_rejected`

These events are visible in:

- request execution timelines
- the live activity stream
- operational debugging flows

## Notes

- The policy is not intended to replace authentication or HTTP rate limiting.
- The policy protects the LLM execution boundary specifically.
- The current implementation is intentionally simple and configurable through environment variables.
